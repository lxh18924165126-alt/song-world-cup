# 项目实施规则

## Applies to / 适用范围

- 用户目标：首次从设计交付包实施 0→1 产品，或跨导入、赛事、离线、分享、导出、登录与后台的完整链路。
- 路径信号：当前的 `README.md`、`CODEX_PROMPT.md`、`docs/`、`mockups/`、`prototype/`，以及未来承载前后端实现的工作区。
- 实际影响：第一版产品范围、前后端完成度、视觉复刻质量、凭证联调和最终交付。

## Does not apply to / 不适用范围

- 不改变产品实现的局部规则治理或纯文案修正；这类任务只加载直接相关规则。
- 只读浏览交付包且不据此作实现决策时，不因路径本身加载完整实施流程。

## Authoritative rules / 权威规则

- 首次 0→1 实施前完整审阅根直接路由的设计交付包；业务行为以 `docs/02_rules/` 为最高优先级，前端结构与视觉以 `mockups/` 和 `prototype/` 为复刻基线，技术与交付约束分别以 `docs/03_technical/` 和 `docs/05_delivery/` 为准。
- 先给出可验证的功能切片计划，再连续实施；计划和切片检查点不是额外用户审批点。
- 第一版必须完整覆盖现有产品文档定义的导入、歌曲检查、赛事设置、抽签、比赛、离线与编辑权冲突、结果、主动赛后分享、本地导出、微信/QQ 登录代码与模拟 Provider、登录迁移和最小后台。
- 前端必须具有真实交互、状态管理和离线逻辑；后端必须具有真实数据模型、迁移与 API。静态页面、假交互、空 API 或占位存储均不算完成。
- 效果图尽量一比一复刻，不得擅自替换为另一套设计；原型只说明结构、层级和关键交互，不能替代生产实现。
- 外部平台凭证缺失时可以完成代码与模拟 Provider，但必须明确标记“代码已完成，生产联调待凭证”，不得声称生产验收通过。
- 若本机代理妨碍 Wrangler 联调或部署，先完成本地实现与验证，将远程步骤放在最后；部署仍须由当前用户目标明确包含。
- 首次完整实现的交付报告应覆盖可运行代码、数据库迁移、实际本地验证、已执行的线上部署及 `workers.dev` 地址、已知问题和降级项；未执行项如实说明。

## Required verification / 必要验证

- 每个功能切片完成后运行最窄直接相关验证；跨层切片同时验证前端、API、持久化与离线边界。
- 按风险逐步覆盖抽签/轮空/锁轮/统计/导出单元测试、主流程集成测试、移动端/离线/接管/分享/导出 E2E、视觉回归和目标浏览器兼容。
- 当前尚无可执行配置，不能预设命令；脚手架落地后使用 manifest 和 CI 中的真实命令，并同步更新根规则及代码阅读说明。

## Prohibitions / 禁止事项

- 不为追求“完整”擅自实现已列入后续版本的能力。
- 不用静态原型的演示数据或浏览器内假响应冒充真实后端、同步或生产联调。
- 不因缺少凭证而跳过可本地实现与验证的代码，也不伪造远程部署或第三方平台结果。

## Evidence / 证据来源

- `CODEX_PROMPT.md`：原始开发提示词及完整迁移来源。
- `docs/00_overview/` 至 `docs/05_delivery/`：产品、业务、技术、设计与验收的权威交付文档。
- `mockups/` 与 `prototype/`：视觉和关键交互基线；后者明确不是生产实现。
