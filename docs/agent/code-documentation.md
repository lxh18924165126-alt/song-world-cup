# 代码功能阅读说明规则

## Applies to / 适用范围

- 用户目标：新增、修改、重构或删除生产功能代码。
- 路径信号：未来的应用、服务、共享包、schema、migration、测试与部署配置，以及 `docs/06_codebase/`。
- 实际影响：功能入口、模块职责、调用链、公共契约、持久化、离线同步、安全边界或验证方式。

## Does not apply to / 不适用范围

- 仅修改效果图、静态原型、产品需求或 Agent 规则，且没有生产代码与实际架构变化。
- 只读查看代码但不改变实现或决策时，不要求制造新的说明文件。

## Authoritative rules / 权威规则

- `docs/06_codebase/` 是代码功能阅读说明的唯一目录；说明必须描述仓库中已经存在的实现，不得把设计设想伪装成现状。
- 每个完成的功能切片在同一改动中新增或更新一份功能说明；重命名、拆分或删除代码时同步更新索引并移除失效说明。
- 说明文件按稳定功能能力命名，而不是按临时任务或工单命名；一个文件可以覆盖一个端到端切片，但必须明确各层边界。
- 每份说明至少记录：功能范围与用户入口、真实代码入口、关键模块及职责、调用/数据流、公共契约与持久化、重要状态和不变量、失败/离线/并发行为、验证路径、相关产品与技术文档。
- 路径、命令、schema、API 和测试必须能在当前仓库复核；避免复制大段源码，优先解释“从哪里读、为什么这样连、改变时还要检查哪里”。
- 首批 manifest、工作区和 CI 落地后，用其中的真实信息补齐根 `AGENTS.md` 的版本/命令来源，并在代码阅读索引记录实际组件边界。

## Required verification / 必要验证

- 检查说明中的每个仓库路径、命令和链接真实存在，且与最终 diff 一致。
- 对照受影响测试和公共契约，确认说明没有遗漏新增入口、数据变化、兼容路径或恢复方式。
- 若生产代码尚不存在，只维护目录索引的“当前状态”，不得创建空功能页。

## Prohibitions / 禁止事项

- 不写 `TODO`、`TBD`、猜测路径、未来类名或尚未落地的技术选型。
- 不用迁移报告、提交历史或过期设计代替当前代码说明。
- 不让说明成为业务规则、schema 或 API 契约的第二份冲突权威；发现不一致时修正文档路由并指向真正权威来源。

## Evidence / 证据来源

- 用户当前明确要求在编码完成后补齐代码功能阅读说明。
- 当前仓库文件清单：只有设计交付物和静态原型，没有 manifest、业务实现、测试或迁移。
